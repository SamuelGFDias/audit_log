**Arquivo:** `src/AuditLog.Generator.SoftDelete/HandlerGenerator.cs`


### Fix 2 — Substituir `ResolveDependentEntityName` por matching semântico

**Arquivo:** `src/AuditLog.Generator.SoftDelete/FluentApiParser.cs`

O método atual (linhas 425-460) usa substring + PascalCase heurística que casa falsos positivos. Substituir por:

1. **Matching exato por navigation property**: se `navProperty` = "CapacitacaoOrgaoInstituicoes" e existe entidade com nome exato `CapacitacaoOrgaoInstituicao` (singular), usar essa.

2. **Matching por FK target name**: se `fkName` = "AreaTecnicaId", extrair "AreaTecnica" (remover "Id"), buscar entidade com esse nome.

3. **Sem fallback por substring/contains**: se nenhum match exato resolver, retornar vazio (a relação não será adicionada como FK referencing).

```csharp
private static string ResolveDependentEntityName(
    string principalEntity, string navProperty, string fkName,
    ImmutableArray<EntityInfo> entities, Dictionary<string, EntityInfo> entitySymbols)
{
    if (entities.Length == 0) return "";
    var candidates = entities.Where(e => e.Name != principalEntity).ToList();
    if (candidates.Count == 0) return "";

    // 1. Match por navigation property (singularizada)
    var navSingular = navProperty.EndsWith("s")
        ? navProperty.Substring(0, navProperty.Length - 1)
        : navProperty;
    var byNav = candidates.FirstOrDefault(e => e.Name == navProperty);
    if (byNav is not null) return byNav.Name;
    var bySingular = candidates.FirstOrDefault(e => e.Name == navSingular);
    if (bySingular is not null) return bySingular.Name;

    // 2. Match por FK target (fkName sem sufixo "Id")
    if (fkName.EndsWith("Id"))
    {
        var fkTarget = fkName.Substring(0, fkName.Length - 2);
        if (!string.IsNullOrEmpty(fkTarget) && fkTarget != principalEntity)
        {
            var byFK = candidates.FirstOrDefault(e => e.Name == fkTarget);
            if (byFK is not null) return byFK.Name;
        }
    }

    return "";
}
```

Remove o bloco `byContains` (linhas 441-457) e o helper `SplitPascalCase`.

### Fix 5 — Tabela associativa: FK deve pertencer ao par correto

**Arquivo:** `src/AuditLog.Generator.SoftDelete/FluentApiParser.cs`

No branch `HasOne` (linhas 274-339), a lógica atual assume que o `principalEntity` no contexto de `HasOne` é o dependente. Isso está correto conceitualmente, mas o problema é que as mesmas FKs também aparecem como `HasMany` do lado principal.

Causa raiz no `ResolveDependentEntityName` Bug #2 — quando a resolução é incorreta, FKs de entidades não relacionadas são atribuídas ao principal errado. Com o Fix 2 implementado, relações como `AreaTecnicaId -> Usuario` não serão mais geradas porque `ResolveDependentEntityName` não vai mais resolver "AreaTecnica" a partir de "Usuario" via substring.

**Fix 5a — Validação extra no config builder:** Em `GetDbContextTarget` (EntityDetector.cs:59-73) ou `GenerateAll` (SoftDeleteGenerator.cs:41-54), verificar que a FK property realmente existe no tipo do dependente:

```csharp
.Where(r => r.PrincipalEntity == e.Name && !r.IsOwnership)
```

Adicionar:
```csharp
// Validar que a FK property existe no dependente
.Where(r => {
    if (r.PrincipalEntity != e.Name || r.IsOwnership) return false;
    // Se o nome da FK não termina com o nome do principal, pode ser relação incorreta
    if (!r.FkPropertyName.StartsWith(e.Name) && !r.FkPropertyName.StartsWith(e.Name + "Id"))
    {
        // Verificar via entityLookup se a propriedade existe
        if (entityLookup.TryGetValue(r.DependentEntityName, out var depInfo))
        {
            var fkProp = depInfo.type.GetMembers()
                .OfType<IPropertySymbol>()
                .FirstOrDefault(p => p.Name == r.FkPropertyName);
            if (fkProp is null) return false;
        }
    }
    return true;
})
```

### Fix 6 — Usar namespace completo em `context.Set<T>()`

**Arquivo:** `src/AuditLog.Generator.SoftDelete/FluentApiParser.cs`

No `HasMany` branch, garantir que `dependentFullName` sempre use `ITypeSymbol.ToDisplayString(SymbolDisplayFormat.FullyQualifiedFormat)` (via `RoslynExtensions.GetFullName`), nunca caia para `dependentEntityName` puro.

```csharp
var dependentFullName = "";
if (!string.IsNullOrEmpty(dependentEntityName))
{
    if (entitySymbols.TryGetValue(dependentEntityName, out var dep))
    {
        dependentFullName = dep.FullName;
    }
    else if (entityLookup.TryGetValue(dependentEntityName, out var depInfo))
    {
        dependentFullName = depInfo.fullName;
    }
    else
    {
        // Fallback seguro: não gerar relação sem namespace
        continue; // ou return empty
    }
}
```

**Arquivo:** `src/AuditLog.Generator.SoftDelete/HandlerGenerator.cs`

Mesma lógica no branch `HasOne` (linhas 309-316) — garantir que `hDepFullName` use `RoslynExtensions.GetFullName` do `ITypeSymbol` do `entityLookup`:

```csharp
var hDepFullName = entitySymbols.TryGetValue(principalEntity, out var hDep)
    ? hDep.FullName
    : (entityLookup.TryGetValue(principalEntity, out var hDepInfo) ? hDepInfo.fullName : "");
```
