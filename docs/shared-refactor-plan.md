# Refatoração: AuditLog.Generator.Shared

## Problema

Atualmente `AuditLog.Generator` (o source generator principal de auditoria) **não usa** o projeto `AuditLog.Generator.Shared`. Apenas `AuditLog.Generator.SoftDelete` linka os arquivos compartilhados via `<Compile Include>`. Isso significa:

- Duplicação potencial de helpers Roslyn/StringBuilder entre os dois generators
- Inconsistência — o generator principal não segue o mesmo padrão do SoftDelete
- Manutenção: se um helper for adicionado ao Shared, o generator principal não o aproveita

## Objetivo

1. Fazer `AuditLog.Generator` linkar `AuditLog.Generator.Shared` da mesma forma que `AuditLog.Generator.SoftDelete`
2. Tornar as classes do Shared `internal` para que não vazem na API pública
3. Garantir que **consumidores do pacote NuGet** nunca tenham acesso aos métodos do Shared

---

## Arquivos a modificar

| # | Arquivo | Ação |
|---|---------|------|
| 1 | `src/AuditLog.Generator.Shared/RoslynExtensions.cs` | Tornar class `internal` |
| 2 | `src/AuditLog.Generator.Shared/SourceProductionContextExtensions.cs` | Tornar class `internal` |
| 3 | `src/AuditLog.Generator.Shared/StringBuilderExtensions.cs` | Tornar class `internal` |
| 4 | `src/AuditLog.Generator/AuditLog.Generator.csproj` | Adicionar `<Compile Include>` link ao Shared |
| 5 | `src/AuditLog.Generator.SoftDelete/AuditLog.Generator.SoftDelete.csproj` | (já tem o link, só verificar) |

---

## 1. Tornar classes do Shared `internal`

Atualmente são `public`. Como os arquivos são compilados dentro de cada generator via file linking, eles precisam ser `internal` para:

- Não vazar como API pública do analyzer
- Evitar conflito se ambos os generators estiverem no mesmo projeto (tipos duplicados)

```diff
- public static class RoslynExtensions
+ internal static class RoslynExtensions
```

```diff
- public static class SourceProductionContextExtensions
+ internal static class SourceProductionContextExtensions
```

```diff
- public static class StringBuilderExtensions
+ internal static class StringBuilderExtensions
```

### Por que `internal` funciona?

Os generators são empacotados como **analyzers** (`analyzers/dotnet/cs`), não como bibliotecas de referência (`lib/`). Mesmo que fossem `public`, o consumidor do NuGet não compila contra eles. Mas `internal` é mais seguro e explícito: "isso não é API pública".

---

## 2. Adicionar link ao Generator principal

Em `src/AuditLog.Generator/AuditLog.Generator.csproj`, adicionar:

```xml
<ItemGroup>
  <Compile Include="..\AuditLog.Generator.Shared\*.cs" Link="Shared\%(Filename)%(Extension)" />
</ItemGroup>
```

Colocar na mesma posição que já existe no SoftDelete — dentro de um `<ItemGroup>` antes do `<ItemGroup>` de pack do analyzer.

---

## 3. Verificar duplicação

Após linkar, pode haver conflitos de compilação se o `AuditLog.Generator` já tiver métodos com o mesmo nome/signature que o Shared.

### Métodos do Shared:

| Método | Existe no Generator principal? |
|--------|-------------------------------|
| `RoslynExtensions.GetFullName()` | ❌ Não |
| `RoslynExtensions.HasAttribute()` | ❌ Não |
| `RoslynExtensions.ImplementsInterface()` | ❌ Não |
| `RoslynExtensions.ExtractPropertyNameFromLambda()` | Não idêntico — Generator tem `ExpressionParser.ExtractPropertyName()` (diferente) |
| `RoslynExtensions.GetCollectionElementType()` | Não idêntico — Generator tem `ResolveCollectionElementType()` (lógica diferente) |
| `SourceProductionContextExtensions.AddHintSource()` | ❌ Não |
| `StringBuilderExtensions.AppendAutoGeneratedHeader()` | ❌ Não (feito inline nos geradores) |
| `StringBuilderExtensions.AppendLine(sb, indent, text)` | ❌ Não |
| `StringBuilderExtensions.AppendIndent()` | ❌ Não |

**Conclusão: não há conflito.** Os métodos do Shared são complementares, não sobrepostos. O generator principal pode começar a usar os helpers do Shared sem quebrar nada.

## 5. Comportamento no NuGet

Quando o pacote `AuditLog.Generator` for publicado:

```
AuditLog.Generator.nupkg
└── analyzers/
    └── dotnet/
        └── cs/
            ├── AuditLog.Generator.dll    ← contém os métodos do Shared (compilados junto)
            └── AuditLog.Generator.Shared.dll  ← ❌ NÃO EXISTE (nunca foi publicado)
```

O Shared não gera DLL própria — ele é código-fonte linkado, compilado **dentro** de cada generator. O consumidor do NuGet:

- **Não** consegue fazer `using AuditLog.Generator;` no código da aplicação (o DLL está em `analyzers/`, não em `lib/`)
- **Não** vê os tipos Shared mesmo se conseguisse (são `internal`)
- O analyzer funciona normalmente, usando os helpers internamente

## Dependências entre planos

```
┌────────────────────────────┐
│ Refatoração Shared (este)  │  ← Executa primeiro
└──────────┬─────────────────┘
           │ (sem conflito,不影响 pack)
           ▼
┌────────────────────────────┐
│ Pack & Publish             │  ← Depois: Directory.Build.props, metadados, script, workflow
└────────────────────────────┘
```

A refatoração não afeta o empacotamento — Shared nunca gera `.nupkg` próprio. Os dois planos podem ser executados em sequência sem bloqueio.
